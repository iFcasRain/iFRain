# HTTP协议 

## 一、HTTP 协议核心定义

HTTP（HyperText Transfer Protocol，超文本传输协议），是**客户端与服务器之间进行数据交互的标准化协议**，基于 TCP/IP 协议，用于传输超文本（HTML、图片、视频、JSON 等），是互联网通信的基础。

核心作用：规定客户端（如浏览器、Python 代码）如何向服务器发送请求，以及服务器如何向客户端返回响应，确保数据传输的标准化、可交互。

## 补充：学习HTTP必备的TCP、IP协议基础

HTTP 基于 TCP/IP 协议族工作（属于应用层协议），学习HTTP无需掌握TCP/IP全部细节，只需掌握与HTTP交互直接相关的核心内容，重点理解“IP负责找地址、TCP负责保可靠”，以下内容足够支撑HTTP学习。

### 一、IP协议基础（核心：找对服务器）

IP（Internet Protocol，互联网协议）的核心作用是**定位网络中的设备**，确保数据能从客户端发送到目标服务器，是HTTP请求能到达服务器的前提。

#### 1. 核心概念（必记）

- **IP地址**：网络中每台设备（电脑、服务器、手机）的唯一标识，类似“家庭住址”，格式分为IPv4（如192.168.1.1）和IPv6（更长，用于解决地址枯竭），服务器必须有公网IP，客户端才能访问。
- **端口**：一台设备上多个服务的“门牌号”，IP定位到设备后，通过端口定位到具体服务。HTTP默认端口是80，HTTPS默认端口是443（对应之前HTTP与HTTPS的区别），比如URL `https://jsonplaceholder.typicode.com/posts/1` 中，默认使用443端口。
- **域名与IP的关系**：域名（如jsonplaceholder.typicode.com）是IP地址的“别名”，因为IP地址难记，通过DNS（域名解析服务）将域名转为IP地址，客户端才能找到服务器（比如输入域名后，DNS解析出服务器IP，再发起HTTP请求）。

#### 2. 与HTTP的关联

HTTP请求的URL（如https://jsonplaceholder.typicode.com/posts/1），本质是通过域名解析出IP地址，再通过端口（默认443）找到服务器上的HTTP服务，完成连接建立。

### 二、TCP协议基础（核心：可靠传输）

TCP（Transmission Control Protocol，传输控制协议）的核心作用是**保证数据在客户端和服务器之间可靠、有序传输**，HTTP的请求和响应数据，都是通过TCP连接传输的，避免数据丢失、乱序。

#### 1. 核心特点（与HTTP关联紧密）

- **面向连接**：传输数据前，必须先建立TCP连接（三次握手）；数据传输完成后，关闭连接（四次挥手），这对应HTTP请求-响应流程的“建立连接→传输数据→关闭连接”。

  三次握手：建立 TCP 连接，3 次交互，确保双方都能收发数据

  四次挥手：关闭 TCP 连接，4 次交互，确保双方都发完数据再断开

- **可靠传输**：数据传输时会确认对方是否收到（确认机制），若数据丢失，会重新发送；同时保证数据按发送顺序到达，避免HTTP请求/响应数据乱序导致解析失败。

- **面向字节流**：将HTTP的请求报文、响应报文拆分成多个字节块传输，到达后再拼接还原，确保大数据（如长JSON、图片）能正常传输。

#### 2. 核心流程（简化版，无需深入底层）

1. **三次握手（建立连接）**：客户端→服务器（请求建立连接）→客户端（确认）→服务器（确认），确保双方都能正常收发数据（对应HTTP请求流程的第一步“建立TCP连接”）。
2. **数据传输**：HTTP的请求报文、响应报文通过建立好的TCP连接传输，TCP保证数据可靠、有序。
3. **四次挥手（关闭连接）**：客户端→服务器（请求关闭）→服务器（确认）→服务器（请求关闭）→客户端（确认），确保数据全部传输完成后再关闭连接（对应HTTP短连接的“关闭连接”步骤）。

#### 3. 与HTTP的关联（重点）

- HTTP的“无连接”（默认短连接），本质是“TCP连接一次请求-响应后就关闭”；HTTP/1.1的长连接（Connection: keep-alive），本质是“TCP连接保持一段时间，复用连接传输多次HTTP请求-响应”，减少连接建立/关闭的开销。
- 若TCP连接建立失败（如服务器IP错误、端口未开放），HTTP请求会直接失败，无法到达服务器（类似“地址错了，快递送不到”）。

### 三、总结（重点记忆）

学习HTTP，只需记住：IP负责“找到服务器”（通过IP地址+端口），TCP负责“把HTTP数据可靠传到服务器/客户端”（通过三次握手、四次挥手、可靠传输机制），两者是HTTP能正常工作的底层基础，无需深入其复杂底层原理。

## 二、HTTP 协议核心特点

1. 无连接：默认情况下，客户端发送请求、服务器返回响应后，TCP 连接立即关闭（HTTP/1.1 支持长连接，通过`Connection: keep-alive`维持）。
2. 无状态：协议本身不记录客户端的历史请求信息，每次请求都是独立的（需通过 Cookie、Session 实现状态保持）。
3. 基于请求 - 响应模型：只有客户端主动发送请求，服务器才会返回响应，服务器不会主动向客户端推送数据。
4. 可扩展：支持自定义请求头、响应头，可结合 HTTPS 实现加密传输，适配不同业务场景。
5. 简单易读：请求和响应格式简洁，采用明文传输（HTTPS 除外），便于调试和理解。

## 三、HTTP 请求 - 响应模型（核心流程）

完整的 HTTP 交互流程，是 API 调用、网页访问的底层逻辑，共 6 步：

1. 客户端（如浏览器、Python 代码）与服务器通过 TCP 协议建立连接（三次握手）；
2. 客户端向服务器发送 HTTP 请求（包含请求方法、请求头、请求体等）；
3. 服务器接收请求，解析请求内容，处理业务逻辑；
4. 服务器向客户端返回 HTTP 响应（包含状态码、响应头、响应体等）；
5. 客户端接收响应，解析响应内容（如渲染网页、提取 JSON 数据）；
6. 若为短连接，TCP 连接关闭（四次挥手）；若为长连接，连接保持，等待下一次请求。

## 四、HTTP 请求报文结构（重点）

请求报文是客户端发送给服务器的 “指令”，由 4 部分组成，顺序固定：**请求行 → 请求头 → 空行 → 请求体**。

### 4.1 请求行（必选）

核心作用：告知服务器「请求方法」「请求地址」「HTTP 版本」，格式如下：

```Plain
请求方法  请求URL  HTTP版本
```

示例：`GET https://www.example.com/api/posts HTTP/1.1`

#### 常用请求方法（高频 4 种）

| 请求方法 | 核心用途        | 特点                                          | 典型场景                         |
| -------- | --------------- | --------------------------------------------- | -------------------------------- |
| GET      | 查询 / 获取资源 | 参数拼在 URL 中，无请求体，可被浏览器直接访问 | API 数据查询、网页访问           |
| POST     | 提交 / 新增资源 | 参数放在请求体中，适合敏感数据、大量数据      | 登录提交、表单提交、API 新增数据 |
| PUT      | 全量更新资源    | 提交完整的新资源，覆盖原有资源                | API 全量修改数据                 |
| DELETE   | 删除资源        | 仅请求删除指定资源，无请求体（或极简请求体）  | API 删除数据                     |

#### 补充说明

- HTTP 版本：主流为 HTTP/1.1（目前最常用），HTTP/2.0（性能更优，支持多路复用），HTTP/3.0（基于 UDP，速度更快）。
- 请求 URL：资源的唯一访问地址，如`https://jsonplaceholder.typicode.com/posts/1`。

### 4.2 请求头（必选）

核心作用：向服务器传递客户端的额外信息（身份、数据格式、鉴权等），由「键值对」组成，常用字段如下：

| 字段名        | 核心作用                                          | 示例                                          |
| ------------- | ------------------------------------------------- | --------------------------------------------- |
| User-Agent    | 标识客户端身份（浏览器 / 代码），避免被服务器拦截 | `User-Agent: Python/3.9 requests/2.31.0`      |
| Content-Type  | 声明请求体的数据格式（核心）                      | `Content-Type: application/json`（JSON 格式） |
| Authorization | 携带鉴权信息（API 调用常用）                      | `Authorization: Bearer abc123456`             |
| Connection    | 控制 TCP 连接状态                                 | `Connection: keep-alive`（长连接）            |
| Host          | 指定服务器域名 / IP，必选字段                     | `Host: www.example.com`                       |

### 4.3 空行（必选）

用于分隔请求头和请求体，是 HTTP 协议的固定格式（即使没有请求体，空行也必须存在）。

### 4.4 请求体（可选）

核心作用：存放客户端向服务器提交的数据，仅 POST、PUT 等方法需要，GET、DELETE 方法通常无请求体。

- 常见格式：JSON（最常用，对应`Content-Type: application/json`）、表单（`application/x-www-form-urlencoded`）、文件（`multipart/form-data`）。
- 示例（JSON 格式请求体）：

```json
{
  "username": "test",
  "password": "123456"
}
```

## 五、HTTP 响应报文结构（重点）

响应报文是服务器返回给客户端的 “结果”，与请求报文对应，由 4 部分组成：**状态行 → 响应头 → 空行 → 响应体**。

### 5.1 状态行（必选）

核心作用：告知客户端「请求处理结果」，格式如下：

```Plain
HTTP版本  状态码  状态描述
```

示例：`HTTP/1.1 200 OK`

#### 核心 HTTP 状态码（必记）

状态码由 3 位数字组成，分为 5 大类，重点记高频码：

1. 2xx（成功）：请求正常处理完成
   1. 200 OK：通用成功（GET/POST 请求成功）
   2. 201 Created：资源创建成功（POST 请求常用）
2. 4xx（客户端错误）：请求存在问题，服务器无法处理
   1. 400 Bad Request：请求参数错误
   2. 401 Unauthorized：鉴权失败（未携带 Token/API Key）
   3. 403 Forbidden：权限不足（已鉴权，但无访问权限）
   4. 404 Not Found：请求的 URL 不存在
3. 5xx（服务器错误）：服务器自身出现问题
   1. 500 Internal Server Error：服务器内部报错
   2. 503 Service Unavailable：服务器暂时不可用（过载 / 维护）
4. 1xx（信息性）：临时响应，告知客户端继续处理（极少用）
5. 3xx（重定向）：请求的资源已移动，需客户端重新发起请求（如 302 临时重定向）

### 5.2 响应头（必选）

核心作用：向客户端传递服务器的额外信息（数据格式、服务器信息、缓存等），常用字段如下：

| 字段名         | 核心作用                            | 示例                                            |
| -------------- | ----------------------------------- | ----------------------------------------------- |
| Content-Type   | 声明响应体的数据格式（核心）        | `Content-Type: application/json; charset=utf-8` |
| Content-Length | 声明响应体的字节长度                | `Content-Length: 128`                           |
| Server         | 标识服务器类型                      | `Server: Nginx/1.21.6`                          |
| Set-Cookie     | 向客户端设置 Cookie（用于状态保持） | `Set-Cookie: sessionid=abc123; Path=/`          |
| Cache-Control  | 控制客户端缓存策略                  | `Cache-Control: max-age=3600`（缓存 1 小时）    |

### 5.3 空行（必选）

分隔响应头和响应体，与请求报文的空行作用一致。

### 5.4 响应体（可选）

核心作用：存放服务器返回给客户端的实际数据（如 HTML 页面、JSON 数据、图片等），是客户端最关心的内容。

- 示例（JSON 格式响应体，API 调用常用）：

```json
{
  "code": 200,
  "msg": "success",
  "data": {
    "id": 1,
    "name": "张三"
  }
}
```

## 六、HTTP 与 HTTPS 的核心区别（高频考点）

| 对比维度 | HTTP                             | HTTPS                                  |
| -------- | -------------------------------- | -------------------------------------- |
| 安全性   | 明文传输，无加密，易被劫持、篡改 | 基于 SSL/TLS 加密传输，安全可靠        |
| 端口     | 默认 80 端口                     | 默认 443 端口                          |
| 证书     | 无需证书                         | 需申请 CA 证书（如阿里云、腾讯云证书） |
| 性能     | 无加密解密过程，性能略高         | 有加密解密过程，性能略低               |
| 用途     | 普通非敏感数据传输（如静态网页） | 敏感数据传输（如登录、支付、API 鉴权） |

## 七、HTTP 常见拓展与补充

### 7.1 长连接与短连接（HTTP/1.1 重点）

- 短连接：每次请求 - 响应后，TCP 连接立即关闭，适合请求频率低的场景（如普通网页访问）。
- 长连接：通过请求头`Connection: keep-alive`维持 TCP 连接，多个请求可复用同一个连接，减少连接建立 / 关闭的开销，适合 API 高频调用、动态网页场景。

### 7.2 缓存机制

HTTP 缓存可减少服务器压力、提升客户端访问速度，核心通过响应头控制：

- `Cache-Control: max-age=3600`：客户端缓存该资源 1 小时，1 小时内无需再次向服务器请求。
- `ETag`：资源的唯一标识，服务器通过对比 ETag 判断资源是否更新，避免重复传输。

### 7.3 跨域问题（API 调用常见）

当客户端（如前端页面）与 API 服务器域名、端口、协议不一致时，会触发跨域限制，核心解决方案：

- 服务器端设置响应头：`Access-Control-Allow-Origin: *`（允许所有域名跨域）。

## 八、学习重点与避坑指南

1. 重点掌握：请求 / 响应报文结构、常用请求方法、核心状态码（200/400/401/404/500）。
2. 避坑点 1：GET 请求无请求体，参数拼在 URL 中，敏感数据（如密码）不能用 GET 请求提交。
3. 避坑点 2：JSON 格式请求 / 响应，需确保`Content-Type`设为`application/json`，否则服务器无法解析。
4. 避坑点 3：状态码 4xx 是客户端问题（如参数错误、鉴权失败），5xx 是服务器问题（需联系服务端排查）。
5. 实操关联：Python API 调用中，`requests`库会自动帮我们构造请求报文、解析响应报文，无需手动拼接格式。
