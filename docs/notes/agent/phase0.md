## 〇、读前说明：这份讲义解决什么问题？

**Phase 0（前置基础）**在学路线里的含义是：

> 你还不需要做出「能自动上网、多工具协作」的完整 Agent，但你要先具备三块能力：  
> **① 理解大模型在干什么** → **② 能用提示词稳定约束输出** → **③ 能用 Python 把输出接到程序里（JSON、异步、密钥）**。

这三块就是后面所有 Agent（工具调用、记忆、RAG、工作流框架）的地基。

## 一、一张「大地图」：AI、机器学习、大模型、Agent

先把关系说清楚（不用背定义，理解层次即可）。

### 1.1 人工智能（AI）

**人工智能（Artificial Intelligence）**是一个很宽的概念：让机器表现出「像人一样解决问题」的能力。可以是规则系统、搜索引擎、推荐系统，也可以是深度学习模型。

### 1.2 机器学习（Machine Learning）

**机器学习**：让机器从数据中学习规律，而不是手写全部规则。  
对你来说，Phase 0 只需要记住一句话：**你调用的大模型，本质是「已经训练好的机器学习模型」**，它的参数和训练过程你不用管（除非以后做算法方向）。

### 1.3 大语言模型（LLM）

**大语言模型（Large Language Model，LLM）**：一种专门处理「文本」的神经网络模型，规模很大（参数多、训练数据多），擅长：

- 续写与改写  
- 摘要、翻译、改写语气  
- 按要求生成结构化文本（在你约束得当时）

**重要直觉**：LLM 本质上是「**概率预测下一小段文字**」叠起来的系统。所以你给它同样的输入，也可能得到略有不同的输出——这就是为什么要有 **温度**、**结构化约束**、**解析与重试**。

### 1.4 Agent（智能体）到底是什么？

在日常工程语境里，**Agent**通常指：

> **LLM +（可选）规划/循环 +（可选）工具调用 +（可选）记忆/RAG +（可选）人机协作与安全边界**  
> 组成的一个「能围绕目标采取行动」的系统。

把它想成：

- **Chatbot（聊天机器人）**：多数时候只是「对话」，不一定能稳定改变外部世界。  
- **Agent**：更像「带职责的助手」，常被设计成能 **调用函数/API**、**分步完成任务**、**在多轮里保持目标**。

Phase 0 还不会做完整 Agent，但你要先理解这些名词，否则 Phase 1 会突然冒出 ReAct、Tools、Memory，你会断层。

### 1.5 Phase 0 在学路线里的位置（对照心智）

```text
Phase 0：你会“调用模型 + 约束输出 + 程序接住输出”
Phase 1：你开始让模型“选工具 + 循环推理”
Phase 2：你用框架管理“状态图 + 持久化”
Phase 3：你对接真实系统（API/MCP/评测）
Phase 4：你做生产级（观测、安全、成本）
```

## 二、Agent 相关「概念词典」（中英对照 + 白话）

下面这些词，你在学 Agent 时**一定会反复遇到**。建议你把这一节当词典，不懂就回来查。

### 2.1 Prompt（提示词）

**Prompt**：你给模型的输入指令与材料。  
更广义地说，一次请求里通常包含：

- **System prompt**：写在 `system` 消息里的「全局规则」。  
- **User prompt**：写在 `user` 消息里的「当前任务」。  

### 2.2 Completion / Chat Completion（补全 / 对话补全）

很多服务商把接口叫 **Chat Completions**：你给一串 `messages`，模型返回一条 **assistant** 文本（或结构化字段）。

### 2.3 Token（词元）

**Token**：模型处理文本的基本单位，不等于「一个字」或「一个词」。  
中文、英文、标点、空格都可能被切成不同 token。

你需要建立的直觉：

- **同样的句子，中英文 token 数量可能差很多**。  
- **计费、速度、上下文上限**，基本都按 token 粗略计量。

### 2.4 Context Window（上下文窗口）

**上下文窗口**：一次请求里模型能「同时看到」的最大 token 总量（上限由模型/服务决定）。

后果：

- 对话太长：早期内容可能被截断或压缩（取决于平台策略）。  
- Prompt 写得过长：浪费额度，还可能让模型更不稳定。

### 2.5 Temperature（温度）

控制采样随机性：**越低越稳，越高越发散**。  
结构化任务（JSON）通常偏低温度。

### 2.6 Top-p（核采样）

另一种控制随机性的参数。Phase 0 可先固定默认，不要同时乱调温度和 top-p。

### 2.7 Messages Roles（消息角色）

常见三类：

| 角色 | 作用（工程上怎么用） |
|------|----------------------|
| `system` | 全局规则：你是谁、输出格式、禁止事项、安全边界 |
| `user` | 本轮任务输入：用户提供的问题/材料 |
| `assistant` | 模型历史回复：多轮对话要把它带回上下文 |

### 2.8 Structured Output（结构化输出）

让模型输出 **可被程序解析** 的格式（最常见是 **JSON**）。  
Agent 工程里结构化输出几乎是「标配能力」，因为下一步往往要：

- 路由到函数  
- 写入数据库  
- 触发工具参数

### 2.9 JSON Schema / 校验（先懂概念）

**JSON Schema**：用一套规则描述 JSON 应该长什么样（字段名、类型、是否必填）。  
Phase 0 你只要知道：**约束越明确，解析越不容易翻车**；具体 schema 写法可以 Phase 3 再强化。

### 2.10 Tool / Function Calling（工具 / 函数调用）（Phase 1 预热）

**工具（Tool）**：模型不直接「编造事实」，而是调用你提供的函数，例如查天气、查数据库。  
**函数调用**：模型输出「要调用哪个函数 + 参数是什么」，由运行时执行并把结果返回给模型。

你现在只需要记住：**Agent 的强大很大程度来自工具**，但 Phase 0 先不学实现。

### 2.11 RAG（检索增强生成）（名词先懂）

**RAG（Retrieval-Augmented Generation）**：先从知识库检索相关内容，再让模型基于检索结果回答。  
Phase 0 你只需要知道：**它是解决幻觉与领域知识不足的常用手段**，详细在 Phase 1。

### 2.12 Memory（记忆）（名词先懂）

常见两类：

- **短期记忆**：当前会话窗口内的消息（会随着变长而被裁剪）。  
- **长期记忆**：向量库、摘要笔记、用户画像等（需要额外工程）。

### 2.13 Hallucination（幻觉）

模型 **自信地生成不真实内容**。  
工程对策包括：工具查证、RAG、引用来源、结构化约束、评测与护栏——Phase 0 先建立概念即可。

### 2.14 Guardrails（护栏）

对输入输出做规则过滤与安全策略（敏感信息、违规内容、注入攻击等）。Phase 4 会更系统。

### 2.15 API Key（密钥）

访问模型服务的凭证。**绝不能**公开提交到 Git、截图、聊天里明文发送。

### 2.16 Rate Limit（限流）

服务商限制单位时间内调用次数。Agent 一旦频繁调用工具/模型，很容易触碰限流，需要重试与降级策略（Phase 3–4）。

### 2.17 Async / Await（异步）

网络请求是 IO：**异步**让你在等待响应时不把整个程序堵死。  
Phase 0 你只要「能读懂、能改参数」，不必深究事件循环原理。

## 三、你要掌握的 6 块内容（0.1～0.6）——展开讲解

---

### 3.1（路线 0.1）Token 与上下文：你到底在付钱买什么？

#### 你要建立的直觉

1. **模型读的不是「字数」，而是 token**。  
2. **上下文窗口是硬上限**：超过就可能失败或被截断。  
3. **Prompt 越长**，剩余的「留给回答」的空间越少（还要看你设置的输出上限）。

#### 和 Agent 的关系（为什么要懂）

Agent 后面要做多轮对话、工具返回结果、RAG 检索片段——这些都会堆进上下文。  
不懂 token/上下文，你会莫名其妙遇到：**模型忘了前面的约束**、**工具结果塞不进**、**费用暴涨**。

#### 实战习惯（Phase 0 就能养成）

- **删掉废话**：礼貌寒暄可以有，但不要重复三遍规则。  
- **规则放 `system`，且短而硬**。  
- **长材料**：先摘要再喂（Phase 1 再系统化）。

#### 自检（口答即可）

- 「上下文窗口」和「模型输出长度上限」是不是一回事？（答案：**不是**；一个是总输入容量相关概念，一个是单次生成长度参数，具体以服务商文档为准，但你要区分两类限制。）

---

### 3.2（路线 0.2）Temperature / Top-p：随机性如何影响「Agent 可靠性」

#### Temperature

- **低温**：更「按部就班」，更像在做考试填空。  
- **高温**：更「天马行空」，更像写作文发散。

对 Agent 的影响：

- **结构化输出（JSON）**：高温会增加字段漂移、格式破坏、额外解释文本。  
- **创意写作**：高温更有趣。

#### Top-p

你可以把它理解为：**从另一个旋钮调节多样性**。Phase 0 建议：**先固定 top-p（常用默认值），只调温度**，避免两个旋钮一起转看不出因果。

#### 复现实验（你可手写记录）

同一个 prompt，连续调用 10 次：

- `temperature=0.1`：统计 JSON 合法次数  
- `temperature=0.9`：统计 JSON 合法次数  

你会得到一个非常直观的结论：**Agent 可靠链路通常要低随机性**。

---

### 3.3（路线 0.3）消息角色：`system` / `user` / `assistant` 的分工

#### 为什么角色重要？

因为服务商的实现通常会：

- 更强地遵循 `system`（相对稳定的全局策略）  
- 把 `user` 当作本轮输入  
- 把历史 `assistant`/`user` 当作对话链条  

Agent 工程里一个很常见的 bug：**把「必须遵守的输出格式」只写在某一轮 `user` 里**，下一轮用户一句话就把它冲淡了。

#### 推荐的写法原则（Phase 0）

1. **硬约束永远放 `system`**：输出只能是 JSON、字段名固定、不要 Markdown。  
2. **任务输入放 `user`**：素材、问题、目标语言等。  
3. **多轮**：只在历史中保留「完成当前任务必要的信息」，不要盲目堆全量日志。

#### 反面教材（你要能识别）

- 「请先解释推理过程，再输出 JSON」（很容易导致输出不再是纯 JSON）  
- system 写 3000 字散文（浪费上下文、稀释关键约束）

---

### 3.4（路线 0.4）结构化输出：Agent 的「插头标准」

#### 为什么 JSON 几乎是第一课？

因为 Agent 后面会：

- 解析「调用哪个工具」  
- 解析工具参数  
- 解析路由决策  

这些都要求：**机器可读**。

#### 你做结构化输出时的「四条铁律」

1. **字段名固定**：例如 `summary`、`translation`。  
2. **类型固定**：字符串就用双引号 JSON；数值不要用 `"123"` 字符串糊弄（除非你明确规定）。  
3. **禁止额外文本**：明确写「不要 Markdown，不要代码块，不要前后缀」。  
4. **代码端必须兜底**：`try/except json.JSONDecodeError`，失败就重试或降级。

#### 最小解析示例（Python）

```python
import json

def safe_parse_model_json(text: str) -> dict:
    text = text.strip()
    try:
        return json.loads(text)
    except json.JSONDecodeError:
        return {}
```

#### 你最常遇到的失败形态（要能辨认）

- 输出：`以下是 JSON：{...}`（前缀破坏解析）  
- 输出：Markdown 代码块包裹 ```json ... ```  
- 输出：键名拼写漂移（`summray`）

对策优先级：**收紧 system > 低温 > 重试一轮「只修正格式」**。

#### 为什么要 `.env`？

- `.gitignore` 忽略 `.env`  
- 代码里读取环境变量，而不是硬编码字符串

### 3.6（路线 0.6）异步与 HTTP：为什么示例代码常出现 `async/await`

#### 直觉解释

调用远程模型是 **网络 IO**：可能要几百毫秒到几秒。

**同步**：线程停在那儿干等（简单但批量时会卡住）。  
**异步**：等待期间可以把控制权交回去（更适合服务和并发）。

Phase 0 你只要做到：

- 看见 `async def`、`await` 不惊慌  
- 知道 **不能在普通函数里随便 `await`**  
- 会复制粘贴 `asyncio.run(main()` 这种入口模式

#### 对照示例（理解即可）

同步：

```python
import requests
r = requests.get("https://example.com", timeout=10)
print(r.status_code)
```

异步：

```python
import asyncio
import httpx

async def main():
    async with httpx.AsyncClient(timeout=10) as client:
        r = await client.get("https://example.com")
        print(r.status_code)

asyncio.run(main()
```
